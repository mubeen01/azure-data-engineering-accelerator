# SQL Naming Standards

These conventions apply to every script under `sql/`. They exist so that scripts, tables,
and code generated by later phases (ADF, Databricks) stay predictable and greppable.

---

## Schemas

The warehouse is organized into layered schemas rather than one schema per business
domain. Industry accelerators (Phase 8) reuse these same schemas — they do not get
their own `healthcare` / `banking` schema.

| Schema    | Purpose                                             | Populated by folder |
|-----------|------------------------------------------------------|----------------------|
| `staging` | Landing zone for raw/incoming data, pre-transform    | 09-seed-data |
| `dim`     | Dimension tables                                     | 03-dimensions |
| `fact`    | Fact tables                                          | 04-facts |
| `etl`     | Stored procedures, functions, watermark/control tables | 06-stored-procedures, 07-functions |
| `audit`   | ETL run logs, data quality/validation results         | 10-validation, 12-maintenance |
| `rpt`     | Reporting views                                       | 05-views |

## Tables

- `dim.dim_<entity>` — e.g. `dim.dim_customer`, `dim.dim_date`
- `fact.fact_<entity>` — e.g. `fact.fact_sales`, `fact.fact_transactions`
- `staging.stg_<source>_<entity>` — e.g. `staging.stg_banking_customers`
- `etl.ctrl_<purpose>` — control/watermark tables, e.g. `etl.ctrl_watermark`
- `audit.log_<purpose>` — e.g. `audit.log_etl_run`, `audit.log_data_quality`

Snake_case, singular schema name, plural avoided in the entity itself (`customer`, not
`customers`).

## Columns

| Pattern | Meaning | Example |
|---|---|---|
| `<entity>_key` | Surrogate key (identity, int/bigint), used for joins | `customer_key` |
| `<entity>_id`  | Natural/business key from the source system | `customer_id` |
| `is_current`   | SCD Type 2 flag for the active row | |
| `effective_date` / `expiry_date` | SCD Type 2 validity window | |
| `created_date` / `created_by` | Audit columns, set on insert | |
| `updated_date` / `updated_by` | Audit columns, set on update | |
| `source_system` | Which upstream system the row came from | |

Every `dim` and `fact` table carries the audit columns above. Every `dim` table
intended for SCD Type 2 carries `is_current`, `effective_date`, `expiry_date`.

## Programmable objects

- Stored procedures: `etl.usp_<verb>_<object>` — e.g. `etl.usp_load_dim_customer`.
  Never use the `sp_` prefix (reserved for SQL Server system procedures).
- Functions: `etl.ufn_<purpose>` — e.g. `etl.ufn_get_current_watermark`.
- Views: `rpt.vw_<purpose>` — e.g. `rpt.vw_sales_summary`.

## Indexes and constraints

- Primary key: `pk_<table>`
- Unique constraint: `uq_<table>_<column(s)>`
- Non-clustered index: `ix_<table>_<column(s)>`
- Foreign key: `fk_<table>_<referenced_table>`

## Files

Scripts are numbered within their folder in execution order:
`<NN>_<Verb>_<Object>.sql`, e.g. `01_Create_Database.sql`, `02_Create_Dim_Customer.sql`.
Numbers are left-padded to two digits and gaps are fine — they don't need to be
contiguous, just ordered.
